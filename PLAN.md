## Статус реализации (Веха 1 — ядро автопереключения)

**Готово и работает:** перехват клавиатуры (пассивно), детекция неверной раскладки
(триграммы RU/EN + словари), исправление через буфер обмена, переключение раскладки,
отмена, горячие клавиши, пользовательские словари, защита команд, доверенный контекст,
защита от порчи текста, CLI, IPC, systemd-сервис, udev/install.

**Окружение:** Ubuntu 26.04, Wayland, GNOME. Код в `/home/alfer/Projects/puntu/`.
Раскладки `us`+`ru`, сочетание переключения `<Super>space`.

### Ключевые отклонения от исходного плана (и почему)

По ходу отладки на реальном GNOME/Wayland выяснилось, что несколько «подтверждённых»
допущений плана не работают. Итоговые решения:

1. **Без `EVIOCGRAB` и ре-эмита — пассивное чтение.**
   Захват клавиатуры + пере-излучение через uinput при любом изъяне **полностью ломал ввод**
   (зависший процесс держал клавиатуру). Перешли на **пассивное** чтение `/dev/input/event*`
   (evdev раздаёт события всем читателям) — мы только *читаем* и *впрыскиваем исправления*.
   Если демон падает/зависает — ввод не страдает. См. `input/capture.rs`, `input/mod.rs`.

2. **Без libxkbcommon — собственные таблицы раскладок.**
   Для фиксированной пары ЙЦУКЕН/QWERTY таблицы в `keymap.rs` точны и убирают системные
   `-dev` зависимости. xkbcommon — апгрейд для произвольных раскладок (позже).

3. **Активную раскладку читаем из `mru-sources[0]`, а не из `current`; и не переключаем сами.**
   Критично: на GNOME `gsettings … current` **не обновляется**, когда раскладку переключает
   Mutter (по `Super+Space`) — застревает на старом значении. Достоверный источник активной
   раскладки — **первый элемент `mru-sources`** (most-recently-used), его Mutter держит свежим.
   Читаем его **заново на каждом завершённом слове** (подпроцесс `gsettings`, не GIO; watcher
   убран как ненадёжный). Из-за неверного чтения `current` ранее ломалось всё: английский «не
   переводился», были ложные правки русского. Авто-переключение раскладки тоже убрано (инъекция
   `Super` открывала «Обзор» GNOME и сбивала фокус) — слова исправляются на месте, раскладку
   пользователь переключает сам. См. `layout.rs::active_lang`.

4. **Исправление — вставкой из буфера обмена, не повтором keycode.**
   Mutter **держит для нашего uinput-устройства фиксированную (us) раскладку** и не меняет её —
   повтор keycode для кириллицы давал латиницу. Поэтому целевой текст кладём в буфер
   (`wl-copy`) и вставляем `Ctrl+V` — это не зависит от раскладки. Прежнее содержимое буфера
   сохраняем и восстанавливаем. Нужен пакет **`wl-clipboard`**. См. `clipboard.rs`, `action.rs`.

5. **Защита от порчи текста (type-ahead guard).**
   Вставка задержана (~десятки мс), и при быстром наборе `backspace` стирал не те символы
   («коверкало» текст). Решение: **не исправлять, если есть необработанный ввод** (look-ahead
   по пакету событий + `poll` дескриптора). Лучше пропустить, чем испортить. Для быстрого
   набора — ручной хоткей. См. `input/capture.rs::has_pending_input`.

6. **Доверенный контекст — мягче, чем в плане.**
   Клик мышью, чтобы сфокусироваться и начать **новое** слово, больше не блокирует исправление.
   Новое слово, набранное подряд, — доверенное; защита от правки середины остаётся (буфер
   очищается при движении курсора посреди слова). См. `buffer.rs`.

7. **Модель — из встроенных словников, не из `.trigrams`.**
   Триграммы строятся при старте из `data/ru.words` (~12 000 слов) и `data/en.words` (~8 000)
   через `include_str!`. Словники — топ частотных слов из открытого списка OpenSubtitles
   (Hermit Dave / FrequencyWords) + немного техслов; чистая кириллица/латиница, длина 2–20.
   `puntu-train` — инспектор оценок, не генератор. Правило решения:
   `score_alt − score_cur > switch_delta(0.7)` **и** `score_alt ≥ alt_valid_min(−3.3)`.
   Чем больше словник — тем выше полнота без потери точности.

8. **Токенизатор: буквы на клавишах пунктуации.**
   Русские буквы `б ж э ю х ъ ё` сидят на EN-клавишах `, ; ' . [ ] \``. Клавиша считается частью
   слова, если она буква **хотя бы в одной** раскладке — иначе слова `будем`/`ужин` рвались.

9. **Зависимости фактические:** `evdev`, `libc`, `serde`/`toml`, `notify`, `clap`,
   `directories`, `signal-hook`, `tracing`, `crossbeam-channel`, `anyhow`/`thiserror`.
   `fst` используется для опционального большого словаря RU (`puntu build-dict` →
   `~/.config/puntu/russian.fst`). **Не используются:** `xkbcommon`, `gio`/`glib`.

### Известные ограничения (следствие пассивной модели на Wayland)
- Автозамена срабатывает, когда после слова есть микропауза; при сплошном быстром наборе
  слова пропускаются (защита от порчи) — на этот случай ручной хоткей `Shift+Pause`.
- Слово, завершённое Enter, в чатах успевает отправиться до исправления.
- Нужен `wl-clipboard`; вставка идёт через `Ctrl+V` (в терминалах это не «вставка» — там
  свой shortcut, поэтому терминалы лучше исключать профилем приложения в M3).
- Качество детекции ограничено размером словников `data/*.words` (расширяемо).

### Не сделано из M1 / отложено
- `dict.rs` (fst-словарь) — не нужен, заменён словниками + userdict.

### Конвертация вставленного текста (`paste_convert`) — реализовано (opt-in)

**Проблема.** Демон читает клавиатуру *пассивно* и копит слово из физических keycode. Текст,
вставленный из буфера (`Ctrl+V`), **не приходит как нажатия клавиш** — его кладёт приложение
через clipboard-протокол. Поэтому буфер слова пуст (а `Ctrl+V` к тому же аккорд → `Invalidate`),
и на пробеле после вставки `buffer.finish()` возвращает `None`, цикл делает `continue` **до**
строки лога → раньше перевода не было и в логах пусто. Это следствие модели, а не баг.

**Решение — триггер «вставка + разделитель».** На `Ctrl+V` запоминаем факт вставки
(`pending_paste`). Если следующая клавиша — разделитель (пробел/таб/энтер), буфер пуст (между
вставкой и пробелом ничего не печатали), аккорд не зажат и демон не на паузе — читаем буфер через
`wl-paste`, прогоняем через `detect::convert_text` (по словам; верно набранные слова и пробелы не
трогаем) и заменяем: backspace на длину вставки + разделитель, затем вставка исправленной формы.
К моменту разделителя `Ctrl` уже отпущен (нет `Ctrl+Backspace`) и вставка успела примениться —
поэтому замена безопасна. Type-ahead guard тот же, что у обычной автозамены. Буфер пользователя
сохраняется/восстанавливается. См. `input/capture.rs::try_paste_convert`, `detect::convert_text`.

**Ограничения.** Срабатывает только на `Ctrl+V` + последующий разделитель (не на средней кнопке /
меню «Вставить»); ориентирован на однострочный текстовый ввод; нетекстовое содержимое буфера не
трогаем. Альтернатива для произвольного выделения остаётся — `Shift+Pause` (`convert_selection`).

**Переключатель `paste_convert` (по умолчанию `false`).** Файловый, hot-reload флаг в
`config.toml` — тем же приёмом, что правки словарей. Меняется **из CLI**
(`puntu config set paste_convert on`) и **из будущего GUI** через общий `Config::load`/`save_to`,
без отдельного протокола поверх сокета. Точки: `config.rs` (поле + `save_to`), `main.rs`
(`puntu config show|set`), `keymap.rs::KEY_V`, `input/{capture,mod}.rs`, `config.example.toml`.

# Исходный утверждённый план (для истории)

# Puntu — аналог Punto Switcher для Ubuntu (Wayland/GNOME)

> Рабочее название: **Puntu** (Punto + Ubuntu). Можно переименовать.

## Контекст

Нужен open-source аналог Яндекс Punto Switcher для Ubuntu: фоновая утилита, которая
следит за вводом и, когда слово явно набрано не в той раскладке (например `ghbdtn`
вместо `привет`), удаляет его, переключает раскладку и печатает заново правильно.
Плюс сопутствующие удобства: ручная конвертация по горячей клавише, отмена,
(позже) автозамена, исправление регистра, индикатор в трее и GUI.

**Окружение (проверено):** Ubuntu 26.04 LTS, сессия **Wayland**, GNOME. `/home/alfer/Projects` —
общий каталог для всех проектов пользователя, поэтому код живёт в **отдельной подпапке
`/home/alfer/Projects/puntu/`**. Rust-тулчейн ещё не установлен. Пользователь **не** входит в
группу `input`. Клавиатура = `/dev/input/event2` (`platform-i8042-serio-0-event-kbd`).

**Главное ограничение — Wayland.** Классический аналог (`xneur`) и большинство гайдов
используют API X11 (XRecord/XTest/xdotool) для глобального перехвата и ввода. **На Wayland
это не работает принципиально** (клиенту не даётся глобальный перехват клавиатуры).
Надёжный путь на будущее — тот же слой, что используют `keyd`, `evremap`, `kmonad`, —
работа на уровне ядра **evdev/uinput**, ниже дисплейного сервера. Это работает одинаково
на Wayland, X11 и в консоли:

- **Перехват:** читаем сырые события из `/dev/input/event*` и делаем `EVIOCGRAB` клавиатуры.
- **Трансляция** keycode+модификаторы → символ через **libxkbcommon** с учётом *активной* раскладки.
- **Переключение раскладки** в GNOME через `gsettings org.gnome.desktop.input-sources current`.
- **Ввод** исправлений через виртуальную клавиатуру **uinput**.

**Решения (согласованы с пользователем):** Rust · первая веха = *ядро автопереключения*
(systemd user service, конфиг TOML, управление через **CLI**; GUI — в M3 поверх того же
бэкенда) · Русский ↔ Английский.

## Принцип работы (исходная идея)

Ключевая идея Punto: *одни и те же физические клавиши* дают разные символы в разных
раскладках. (Реализация: см. раздел «Статус» выше — повтор keycode заменён вставкой из буфера.)

1. Перехват + прозрачный ре-эмит через uinput. *(отклонено → пассивное чтение)*
2. Анализ по словам: копим keycode, транслируем в текст активной раскладки.
3. Детекция: сравниваем строку в текущей раскладке против тех же keycode в другой.
4. Исправление: backspace → переключить раскладку → повторить keycode. *(отклонено → буфер+Ctrl+V)*
5. Стек отмены + обучение «не срабатывать».

## Алгоритм детекции
- Токенизатор: буквы продлевают слово; пробел/пунктуация закрывают.
- Два кандидата: `S_cur` (текущая раскладка) и `S_alt` (другая).
- Скоринг триграммами: срабатываем, когда другая раскладка уверенно лучше и текущая невалидна.
- Защита: пользовательский словарь (исключения ∪ learned), список команд + эвристики,
  список force (всегда конвертировать).

## Корректное поведение при редактировании
Доверенный контекст: автокоррекция только когда слово набрано подряд и курсор в его конце;
навигация/правка/клик помечают контекст недоверенным. *(Уточнено: новое слово после клика —
доверенное; см. «Статус».)*

## Пользовательские словари, команды, сленг и обучение
Списки: исключения (manual), авто-обученные (learned), исправления (force), команды (commands).
Хранение: `~/.config/puntu/*.txt`, hot-reload. Хоткеи «в исключения» / «исправить+в исправления».
CLI: `puntu dict list/add/rm/forget/clear-learned`.

## Права доступа и запуск (без root)
systemd user service; группа `input` (чтение клавиатуры); udev-правило + модуль `uinput`
(запись). Приватность: локально, без сети/телеметрии; дневник исключён из MVP.

## Улучшения относительно оригинала
Wayland-нативность; приватность; статистическая детекция + управляемый словарь; свои хоткеи;
профили приложений (M3); hot-reload TOML; CLI+UI поверх одного бэкенда; авто-защита команд;
корректная точечная правка.

## Декомпозиция (Вехи)
- Этап 0: окружение (rustup, структура, права).
- Этап 1: перехват (devices/capture/emitter).
- Этап 2: раскладка + keymap.
- Этап 3: буфер + детектор + словари + `--stdin` харнесс.
- Этап 4: исправление + отмена + хоткеи.
- Этап 5: конфиг + IPC + CLI + упаковка.

## Будущие фазы
- M2: автозамена/расширение текста, исправление регистра, конвертация выделения.
- M3: индикатор GNOME + настройки GTK4/libadwaita, профили приложений.
- M4: больше раскладок (казахская), ML-детектор, звуки, opt-in дневник.

## M5 — Миграция на IBus engine (фундаментальное решение race conditions)

### Проблема текущего подхода

М1-М4 используют **uinput-инжекцию keystrokes** для коррекции: backspace удаляет неверное
слово, clipboard+Ctrl+V вставляет правильное. Это даёт **неустранимые гонки** при быстром
вводе:

1. **Pre-grab type-ahead**: между detection separator и активацией keyboard grab проходит
   ~30-100мс (wait_key_release). За это время user'овские нажатия попадают compositor → app
   напрямую. Расширение backspace count + replay через uinput частично решает, но создаёт
   следующую гонку.
2. **Replay vs typing race**: после ungrab мы replay'им user keystrokes через uinput,
   параллельно user продолжает физически печатать → события приходят в app в неправильном
   порядке (`gпривет`, `fработаете`, `sвырработаете`).
3. **350мс debounce** (текущее решение) устраняет гонки ценой того что **коррекция
   срабатывает только при паузе**. При непрерывной печати — слова остаются в wrong-layout.

Wayland `zwp_input_method_v2` на GNOME **недоступен** для внешних клиентов — Mutter
зарезервировал его для IBus как единственный input-method framework. Поэтому правильный
архитектурный путь = **реализовать IBus engine**.

### Архитектура IBus engine

IBus engine — это DBus-сервис, который IBus регистрирует как один из движков ввода. User
выбирает его через GNOME Settings → Keyboard → Input Sources. Когда движок активен, **все
keystrokes пользователя в текущем фокусе проходят через нас перед попаданием в app**.

```
┌─────────┐  keystroke    ┌──────┐  ProcessKeyEvent  ┌──────────────┐
│  user   │ ───────────→ │ IBus │ ────────────────→  │ puntu-engine │
│ kbd     │              │ daemon│                    │   (Rust)     │
└─────────┘              └──────┘ ←────────────────  └──────────────┘
                          │       commit_text/forward
                          ▼
                        ┌──────┐
                        │ app  │
                        └──────┘
```

**Поток обработки слова:**
1. User нажимает `g`, `h`, `b`, `d`, `t`, `n` → IBus передаёт каждое в `ProcessKeyEvent`.
2. Мы **возвращаем true** (handled) для каждой буквы и копим их в внутреннем word buffer.
   App пока не видит ничего — символы наши.
3. User нажимает SPACE → детектор решает Convert.
4. Мы вызываем `commit_text("привет ")` — IBus вставляет текст в app атомарно. Никакого
   backspace, никакого clipboard, никакого race.
5. Если detector решил Leave — `forward_key_event` для каждого keystroke (передать в app
   как есть).

**Преимущества:**
- Мгновенная коррекция (нет debounce)
- Нет гонок (мы единственный писатель в app)
- Layout-independent (commit_text принимает Unicode-текст)
- Корректно работает с любым приложением (text fields, terminals, IDE)

**Сложности:**
- IBus engine для wrong-layout детекции — нетипичный use-case (обычно engines делают CJK
  composition). Возможны edge-cases с `forward_key_event` для пропущенных слов.
- Live preedit: пока user печатает слово, app ничего не видит. Хорошо это или плохо — нужно
  тестировать с реальными приложениями (curses-апп типа vim может не любить).
- Совместное существование с другими engines (например, ibus-typing-booster).

### Этапы реализации

**M5.1 — Каркас IBus engine**
- Добавить зависимости: `zbus = "4"`, `serde` для конфига (уже есть).
- Создать `src/ibus.rs` с DBus-сервисом, реализующим IBus engine interface (`org.freedesktop.IBus.Engine`).
- Component XML файл `data/puntu.xml` для регистрации в IBus.
- `install.sh` копирует `puntu.xml` в `~/.local/share/ibus/component/` + cmd для запуска
  `puntu --ibus` как service.
- Bin target `puntu-ibus` (новый, отдельный от uinput-демона).

**M5.2 — Базовая обработка keystrokes**
- ProcessKeyEvent → классификация через существующий `keymap::classify`.
- Word buffer — переиспользуем `buffer::WordBuffer`.
- На separator → detector.decide() → commit_text(alt) или forward_key_event(original).
- Backspace в word buffer → корректно update preedit.

**M5.3 — Preedit (визуализация набираемого слова)**
- Через `update_preedit_text` показывать пользователю что мы накапливаем буквы.
- На separator — committed text появляется в app, preedit очищается.
- Тестировать на gedit, firefox, gnome-terminal, vim.

**M5.4 — Hotkeys и manual triggers**
- Pause/Break → undo last commit. Для undo нужно знать что было committed; можно через
  `delete_surrounding_text` + commit оригинала.
- Convert selection (Ctrl+Shift) — через `surrounding_text` API получить текущее выделение,
  заменить.

**M5.5 — Coexistence с uinput-демоном**
- Сохранить uinput-демон как **fallback** для приложений где IBus не работает (например,
  legacy X11 apps без XIM).
- CLI флаг `puntu --mode={ibus,uinput,auto}`. По умолчанию `auto` — определить по
  `XDG_SESSION_TYPE`.
- Один shared core (buffer, detector, dict, config); только frontend разный.

**M5.6 — Distribution**
- `install.sh` определяет GNOME → ставит IBus engine. Не-GNOME (sway, hyprland) → uinput.
- Документация в README: «GNOME: пользуйтесь IBus engine, переключите в Settings…».

### Оценка
~3-7 дней разработки (по 4-6 часов), плюс ~2-3 дня тестирования с разными приложениями.
Главный риск: GNOME Mutter может не поддерживать `forward_key_event` так как мы ожидаем,
либо preedit может конфликтовать с native input. Мини-PoC (M5.1+M5.2 за 1 день) даст
понимание.

### Известные альтернативы (отвергнуты)
- **zwp_input_method_v2 напрямую через wayland**: на GNOME недоступно для не-IBus клиентов.
- **GNOME Shell extension через mutter API**: язык JS, не Rust; работает только в GNOME.
- **Полностью отказаться от auto-correct**: лишает проект главной фичи (Punto-style).
